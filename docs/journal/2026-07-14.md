## Red-team round 2 — sibling-product parity (#198 → PR #199, merged)

nene-records completed the same assessment on 07-13 and found two *real*,
exploited types: **F-01** unauthenticated read of the admin API (webhook
secret / drafts) and **F-02** cross-tenant read by replaying a JWT against
another org's host. Today's round verified, by live fire against a disposable
Docker stack, that **neither type is present in Vault**:

| nene-records finding | Vault verdict | Evidence |
|---|---|---|
| F-01 unauth admin GET (Critical) | NOT PRESENT | all 11 admin GET endpoints → 401 unauthenticated (blocklist auth, fail-closed, #157) |
| F-02 cross-tenant JWT-vs-host (Medium) | NOT PRESENT | org-A JWT + org-B `Host` → org-A data only; the signed `org` claim binds the tenant over the host (#141) |
| F-03 version-disclosure headers (Low) | already fixed round 1 | `Server: Apache`, no `X-Powered-By` |

The structural reasons are Vault's blocklist authentication and claim-based
tenant resolution. To remove the *latent* fragility that maps to the sibling's
F-02 root cause, `CapabilityMiddleware`'s organization-scope check was made
**unconditional** — it now runs for every authenticated org-scoped request
regardless of whether the route maps to a capability, so a future admin route
with no capability mapping stays tenant-bound instead of skipping the check.
Not live-exploitable (claim resolution + repo scoping already protected every
route); classified honestly as defense-in-depth. Pinned by
`tests/Auth/CapabilityMiddlewareTest.php` (6 tests).

Round 1's probe had also exercised CSV formula-injection via `GET
/admin/vault/export` — a POST-only route, so the GET returned no CSV and the
check passed vacuously. Corrected to `POST … {"format":"csv"}`; the real
manifest now shows the user cell `=2+5+cmd|calc` neutralized to `'=2+5+cmd|calc`
by `CsvWriter` (ADR 0015). Added `docs/security/harness/probe-tenant.sh` (a
self-contained subdomain-mode host-reuse test) and a `BASE_DOMAIN` passthrough
in `docker-compose.yml`. Main probe `PASS=55 VULN=0 INFO=0`, tenant probe 4/4,
`composer check` green (400 tests). Report:
`docs/security/2026-07-14-redteam-assessment.md`.

## Production installer removed, then made self-removing (board B → #200 / PR #201)

The 07-12 deploy hand-off flagged `install.php` / `installer.js` still sitting
in the production docroot since the 07-10 build (never web-run, so its
success-path self-unlink never fired). Exposure was limited — `.installed`
marker present, so the `ReInstallationGuard` served the "blocked" view and
refused setup — but leaving an installer in the docroot is poor hygiene.

Aligning with the sibling Rinas' SSH deploy convention (backup → minimal
change → verify → report): backed both files up to
`~/backups/vault-installer-removed-20260714/` on the host, removed them, and
confirmed `.env` / `storage` / `var` untouched. Post-removal `/install.php`
serves the SPA shell (identical to any unmatched path) with no installer
content — the attack surface is gone.

The recurrence fix (#200 / PR #201): the self-unlink only ran on the web
success path and only removed `install.php`. Now `remove_installer_files()`
removes **both** files, and `refuse_install()` calls it whenever the
re-installation guard fires — so a deploy/CLI-provisioned install (which never
runs the web step, but writes the marker) self-cleans on the first hit to
`install.php`. Verified end-to-end in a writable temp docroot via `php -S`:
installed → 403 + both files unlinked; not installed → 200 wizard + both files
preserved (no over-deletion).
