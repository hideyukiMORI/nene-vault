## Compliance lock-in: frontend-ci always-on + job-name uniqueness (#246 → PR #247, merged)

Phase 0-1b [reported: hub, owner-delegated]: fleet-wide, `main` branches got
`required_status_checks` locked with `enforce_admins: true` — CI-red merges are
now physically refused [reported]. vault's `frontend-ci.yml` still carried a
`paths: frontend/**` filter, so docs/backend-only PRs skipped the frontend
check and the required `check` context was not fully effective. Removed the
paths filter (both `push` and `pull_request`) so frontend CI runs on every PR
like backend, and renamed the frontend job `check` → `frontend-check` to end
the same-name context collision with backend's `check` job.

Deliberately **kept backend's job named `check`**: renaming the job that
produces the *required* context would leave `check` permanently Expected and
block the PR itself under `enforce_admins` — a deadlock deal hit in the field
the same night [reported]. Keeping backend as `check` meant the required
context stayed satisfied and the PR self-merged (both `check` 40s and
`frontend-check` 1m6s green [measured]); merged to `6384381`. hub then **ADD**'d
`frontend-check` to the required contexts (now `check` + `frontend-check`)
[reported] — an add, not a rename, so no block.
